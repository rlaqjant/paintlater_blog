## 블로그 포스트 작성 워크플로우 (필수 규칙)

PaintLater 서비스 소스 코드를 다루는 블로그 포스트를 작성할 때는 **아래 순서를 반드시 따른다**.

### 1단계 — 정밀 분석

포스트 주제가 정해지면, 먼저 `oh-my-claudecode:architect` 에이전트(Opus, READ-ONLY)로 대상 저장소의 관련 코드를 정밀 분석한다.

**분석 대상 저장소 (절대 경로)**

- 프론트엔드: `/Users/burns/Works/projects/miniature-backlog/miniature-backlog-web`
- 백엔드: `/Users/burns/Works/projects/miniature-backlog/miniature-backlog-api`

**에이전트 호출 원칙**

- READ-ONLY 에이전트를 사용한다 (원본 저장소 변경 방지).
- 필요 시 `Explore` (very thorough) 로 파일 좌표를 먼저 수집한 뒤 `architect` 에 전달한다.
- 프롬프트에는 반드시 "코드에 없는 내용은 '확인 필요'로 명시할 것", "추측 금지" 를 포함시킨다.
- 한 번의 분석은 **하나의 포스트 범위로 좁힌다**. 저장소 전체 분석은 금지.

### 2단계 — 분석 결과 문서 작성 (선행 필수)

블로그 포스트 본문을 쓰기 **전에**, 분석 결과를 반드시 `docs/` 경로 아래에 마크다운 파일로 먼저 작성한다.

**파일 명명 규칙**

```
docs/analysis/<YYYY-MM-DD>-<slug>.md
```

예시:
- `docs/analysis/2026-04-20-jwt-token-queue.md`
- `docs/analysis/2026-04-25-delta-e-paint-matching.md`
- `docs/analysis/2026-05-02-capacitor-oauth-deeplink.md`

**분석 문서 필수 항목**

1. **주제 및 관련 포스트 slug** — 어떤 블로그 포스트의 기반 자료인지 명시.
2. **분석 대상 파일 목록** — 저장소 루트 기준 상대 경로 + 주요 라인 번호.
3. **기술 스택 / 버전** — `package.json`, `build.gradle` 등에서 직접 확인한 수치만.
4. **흐름 정리** — 정상 케이스, 경합/실패 케이스, 엣지 케이스를 분리해 서술.
5. **주요 상수 / 설정값** — 하드코딩된 임계값(예: `OWNED_PREFER_THRESHOLD = 5.0`) 을 파일:라인과 함께 발췌.
6. **설계 의도 / 트레이드오프** — 코드에서 읽어낼 수 있는 범위에서만 서술. 추측은 별도 섹션으로 분리.
7. **확인 필요 영역** — 본 분석에서 다루지 못한 영역을 명시 (할루시네이션 방지).
8. **블로그 포스트 구조 초안** — 문제 상황 → 분석 → 해결 → 교훈의 구조로 섹션 뼈대만 정리.

### 3단계 — 포스트 작성

분석 문서가 존재하고 검증이 끝난 후에야 `content/posts/` 에 실제 MDX 포스트를 작성한다.

**포스트 작성 시 준수 사항**

- 분석 문서에 기록된 **파일:라인 기준으로만** 코드 인용. 분석 문서에 없는 라인은 인용 금지.
- 분석 문서의 "확인 필요 영역" 에 해당하는 내용은 포스트에 쓰지 않는다. 필요하면 다시 분석 단계로 돌아간다.
- 포스트 작성 완료 후, 해당 분석 문서의 상단에 "→ `content/posts/<slug>.mdx` 에 반영됨" 링크를 추가한다.

### 금지 사항

- ❌ 분석 문서 없이 포스트 본문을 먼저 쓰기
- ❌ 에이전트 분석 결과를 그대로 포스트에 복사 (재작성 필수)
- ❌ "대략 이럴 것이다" 식 추측 서술
- ❌ 한 분석 문서에 여러 포스트 주제를 섞기
