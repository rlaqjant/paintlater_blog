# PaintLater 블로그 기본 문체 가이드

> **적용 범위**: `content/posts/**/*.mdx` 에 작성되는 모든 블로그 포스트.
> **기반 자료**: `docs/style/ly-tone-study.md` — LY Corporation Tech Blog 한국어 포스트 5편 분석.
> **원칙**: 이 가이드는 **문체에 한정**한다. 사실 관계 / 코드 인용 / 파일 좌표 규칙은 `CLAUDE.md` 의 블로그 작성 워크플로를 따른다.

---

## 0. TL;DR — 한 문단 요약

합쇼체(`~합니다`)로 일관한다. 1인칭은 `저 / 저희`, 반말·해요체·이모지 금지. 도입부는 인사 + 글의 범위 예고, 본문은 문제 → 원인 → 대안 → 구현 → 결과(숫자) → 교훈 순서. 마무리 섹션 제목은 **`마치며`** 로 고정하고, 독자에게 "비슷한 고민을 하는 분께 도움이 되기를 바랍니다"로 닫는다. 숫자는 적극 사용하되 전부 파일:라인 또는 로그 근거에서만 인용한다.

---

## 1. 목소리(Voice)

- **성격**: 차분한 엔지니어. 과장 없음. 결론을 먼저 밝히되 근거를 꼼꼼히 적는다.
- **거리감**: 독자와는 정중한 거리. 친근하지만 가볍지 않음.
- **겸손**: 해결책을 제시한 뒤에도 "아직 ~ 단계입니다", "~의 경우에는 다를 수 있습니다" 같은 단서를 붙여 **완결성을 주장하지 않는다**.
- **정직**: 실패와 시행착오를 감추지 않는다. "저도 처음에는 ~로 접근했다가 막혔습니다" 같은 서술을 환영한다.

## 2. 어미 / 문장 종결

| 상황 | 기본형 | 예 |
|------|--------|-----|
| 서술 | `~합니다 / ~했습니다 / ~됩니다 / ~입니다` | "JWT 만료는 API 게이트웨이에서 먼저 걸립니다." |
| 바람 / 권유 | `~바랍니다 / ~드립니다 / ~추천합니다` | "같은 문제를 겪는 분들께 도움이 되기를 바랍니다." |
| 부드러운 연결 | `~는데요 / ~인데요` (과하지 않게) | "대표적인 라이브러리로 A, B, C가 있는데요." |
| 속마음 인용 | 작은따옴표 + `~일까?` / `~어야 하나?` | "‘이대로 릴리스해도 괜찮을까?’ 라는 불안이 남았습니다." |

**금지**:

- ❌ 반말 / 해요체 (`~야`, `~지`, `~해요`, `~이에요`)
- ❌ 하오체, 하십시오체와 합쇼체 혼용
- ❌ `~인 것 같습니다` 남용 (확정할 수 있는 사실이면 확정해서 서술)
- ❌ `~것이다` 남발

## 3. 인칭 / 호칭

- 글쓴이 = **`저`**
- 서비스 / 작업 단위 = **`저희 서비스`** 또는 **`PaintLater`** 직접 호명. PaintLater는 현재 1인 개발 구조이므로 팀 수사(`저희 팀`, `전사`)는 사용하지 않는다.
- 독자 호칭은 **`여러분`** 을 도입부 질문에서 한 번 정도만 쓰고, 본문에서는 직접 부르지 않는다.
- 글쓴이가 여럿일 경우에는 `저희` 허용. 공동 저자는 도입부에 이름을 함께 적는다.

## 4. 어휘 / 용어 규칙

### 4.1 영어 기술 용어

**최초 등장 시 원어 + 한국어 풀이 또는 원어 + 한자/외래어 병기**. 이후에는 원어 또는 한국어 단일 표기.

- ✅ `단일 장애점(single point of failure, SPOF)`
- ✅ `피처(feature)`, `스키마(schema)`, `컨텍스트 스위칭`
- ✅ `RAG(retrieval-augmented generation)`

### 4.2 제품명 / 고유명사

**원어 그대로**. 한글 음차 금지.

- ✅ `Next.js`, `Supabase`, `Capacitor`, `Vitest`, `Turborepo`
- ❌ `넥스트제이에스`, `슈파베이스`

### 4.3 생활화된 외래어

한국어 표기가 업계 표준으로 정착된 용어는 한국어로.

- ✅ `배포`, `릴리스`, `롤백`, `파이프라인`, `리뷰`, `토큰`, `쿼리`

### 4.4 번역투 척결

| 금지 | 교정 |
|------|------|
| `~에 대해서 / ~에 관하여` | `~을 / ~를` |
| `~을 가지고 있습니다` | `~이 있습니다 / ~을 보유합니다` |
| `~에 의해 처리됩니다` | 능동문 또는 `~로 처리합니다` |
| `~되어지다 / ~하여진` (이중 피동) | 단일 피동 또는 능동 |
| `~할 수 있을 것이다` | `~할 수 있습니다` |
| `~하는 것이 가능합니다` | `~할 수 있습니다` |

## 5. 문장 리듬

- **평균 길이**: 중~장문. 접속 어미(`~면서`, `~기에`, `~기 때문에`, `~고`, `~지만`)로 두세 절을 이어 붙이되, 핵심 주장 앞뒤에서는 짧게 끊어 강조한다.
- **나쁜 예**: "~이고, ~이며, ~이지만, ~이어서 …" 3단 이상 중첩.
- **좋은 예**: 긴 문장으로 맥락을 설명한 뒤 한 문장으로 단언. "결국 문제는 두 가지였습니다."
- **한 문단 = 하나의 주장**. 여러 주장이 섞이면 문단을 쪼갠다.

## 6. 감정 / 수사

- 감정 서술은 **가끔, 정확하게**. 가슴이 답답했다 → 맥락 없이 쓰면 값싼 감정. 실제 불안의 근거를 먼저 쓰고 한 문장으로만 털어놓는다.
- 다음은 허용되는 감정·상태 어휘의 범위다.
  - `통감했습니다`, `몸소 경험했습니다`, `실감했습니다`, `불안했습니다`, `아쉬움이 남았습니다`
- **금지** 수사:
  - `그야말로`, `정말이지`, `무척이나`, `엄청`, `너무너무`
  - 과장 형용("혁신적인", "압도적인", "완벽한") — 결과 수치로 대체한다.

## 7. 부호 / 서식

- **작은따옴표 `' '`**: 속마음 인용, 용어 강조, 살짝 비틀어 쓰는 표현.
- **큰따옴표 `" "`**: 실제 발화 / 사용자 문의 / 로그 메시지 인용.
- **백틱 `` ` ` ``**: 코드 식별자, 파일 경로, 명령어.
- **이모지**: 금지. 섹션 제목 / 본문 / 목록 어디에도 사용하지 않는다.
- **굵게 `**bold**`**: 본문에서 아껴 사용한다. 한 섹션당 1~2회 이내.
- **기울임 `*italic*`**: 한국어 본문에서는 가독성이 떨어지므로 원칙적으로 피한다.

## 8. 숫자 / 근거

- 구체 수치는 **반드시 소스 근거**(파일:라인, 로그, 벤치마크 스크립트)를 동반한다. 근거 없는 "대략 2배 빨라졌다" 류 서술 금지.
- 대외비로 공개할 수 없는 수치는 **공개하지 않되 이유를 한 줄 밝힌다**.
  > "정확한 트래픽은 공개하기 어렵지만, 상대적 비율로 보면 도입 전후로 약 3.2배 차이가 납니다."
- 퍼센트와 배수는 **기준점**을 함께 쓴다. "40% 절감" 만으로는 부족. "기존 YARN 환경 대비 40% 절감" 까지 적는다.

## 9. 코드 인용

- 언어 지정 펜스 필수: ` ```ts `, ` ```sql `, ` ```bash `.
- 코드 블록 앞 한 문장으로 **무엇을 보여주는 코드인지** 명시한다.
- 긴 코드는 핵심 부분만 남기고 `// ...` 로 생략. 생략한 부분의 역할을 한 줄로 보완한다.
- 코드 인용 위치는 분석 문서(`docs/analysis/*.md`) 에 기록된 **파일:라인 기준으로만**. 이 규칙은 `CLAUDE.md` 에서 내려온 의무 사항이다.

## 10. 섹션 구조 템플릿

```
# <제목>

(도입: 인사 + 자기소개 + 이 글의 범위. 3~5문장.)

## 들어가며: <부제>
(문제 상황을 감각적으로 묘사. 당시 맥락, 역할, 수치 일부.)

## <문제명사>
(원인 분해 2~4개. 소제목으로 분리 또는 번호 목록.)

## <대안 검토 / 선택 근거>
(고려한 대안, 버린 이유, 택한 이유.)

## <구현>
(코드 + 주요 상수 + 도식. 파일:라인 근거 필수.)

## <결과>
(정량 지표. 기준점 명시. 대외비는 이유와 함께 마스킹.)

## 교훈
(번호 매긴 2~4개. "첫째는 ~입니다" 패턴.)

## 마치며
(향후 계획 한 문단 + 독자에게 바라는 한 문장.)
```

### 10.1 섹션 제목 스타일

- **질문형**: "왜 JWT 큐가 필요했을까요?"
- **명사 요약형**: "순환 의존 딜레마"
- **부제 결합형(콜론)**: "들어가며: 쌓여 가는 재시도 로그"
- **마무리 섹션 고정**: **`마치며`** 또는 **`마치며: <부제>`**

## 11. 도입부 / 마무리 관용구

### 11.1 도입부 예시

> "안녕하세요, PaintLater에서 AI 도색 파이프라인을 만들고 있는 [이름] 입니다. 이 글에서는 ~를 해결하기 위해 시도했던 ~과정과 그 결과를 공유하고자 합니다."

> "이 글에서는 먼저 문제 상황을 정리한 뒤, 왜 ~방식을 택했는지, 그리고 실제 도입 후 어떤 변화가 있었는지 순서대로 다룹니다."

### 11.2 마무리 예시

> "비슷한 고민을 하고 계신 분들께 이 글이 참고가 되기를 바라며 글을 마치겠습니다. 긴 글 읽어 주셔서 감사드립니다."

> "아직은 ~ 단계이지만, 이 방향이 작게 시작해 꾸준히 개선해 나가기에 적합하다고 믿습니다. 다음 단계에서는 ~을 시도해 볼 계획입니다."

### 11.3 저자 Bio 포맷

```
Name: [이름]
Description: PaintLater에서 [역할]을 맡고 있습니다. [한 줄 관심사 또는 철학 - 선택]
```

## 12. 금지 목록 요약

- ❌ 이모지 전면 금지
- ❌ 반말 / 해요체 / 하오체 혼용
- ❌ 과장 수사 (`혁신적`, `완벽한`, `압도적`, `그야말로`)
- ❌ 근거 없는 수치
- ❌ 번역투 (`~에 대해`, `~을 가지고 있습니다`, `~되어지다`)
- ❌ 한 문단에 여러 주장 섞기
- ❌ 한국어 본문에서 영어 문장 방치 (원어 + 한국어 풀이 필수)
- ❌ `~것 같습니다` 남용 (확정 사실은 확정해서 쓴다)

## 13. 체크리스트 (포스트 발행 직전)

- [ ] 합쇼체로 일관되는가 (어미 검사)
- [ ] `저 / 저희 서비스` 로 통일되었는가
- [ ] 이모지가 하나도 없는가
- [ ] 영어 용어 최초 등장 위치에 병기가 있는가
- [ ] 수치 / 로그 / 코드 인용이 모두 분석 문서에 기록된 파일:라인에 대응하는가
- [ ] 섹션 마지막이 **`마치며`** 로 끝나는가
- [ ] 도입부 3~5문장에 인사 + 글의 범위가 들어갔는가
- [ ] 문단 하나당 주장 하나인가
- [ ] 번역투(`~에 대해`, `~을 가지고 있다`, `~되어지다`) 검색 결과가 0인가
- [ ] 과장 수사(`혁신적`, `완벽한`, `압도적`)가 없는가
