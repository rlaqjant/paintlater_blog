Spring Boot에서 `@RestController`를 붙이고 `@GetMapping`만 달아도 REST API는 동작합니다. 튜토리얼 한 편이면 CRUD가 돌아가고, Postman에서 200이 돌아오는 것을 확인하면 일이 끝난 것 같습니다.

그런데 API가 여러 도메인으로 커지고, 프론트와 모바일 앱이 동시에 붙기 시작하면 이야기가 달라집니다. 누군가는 URL에 `getUsers` 같은 동사를 박고, 누군가는 200에 `{ success: false }`를 담아 내려보냅니다. 에러 응답이 엔드포인트마다 모양이 다르고, 심한 경우 JPA 엔티티가 그대로 직렬화돼서 `Hibernate.BYTE[]_SENTINEL`이 클라이언트까지 튀어나가기도 합니다.

이 글은 PaintLater 백엔드(Spring Boot 4.0.1, Java 17)가 이 세 가지를 **규약 레벨로 박아놓은 방식**을 정리한 기록입니다. 세 축으로 묶어 봅니다.

1. **리소스와 URL 설계** — 어떤 이름을 어디에 붙일 것인가
2. **상태 코드와 에러 응답** — 실패를 어떻게 표현할 것인가
3. **DTO 경계** — 엔티티와 API 계약을 어떻게 분리할 것인가

각 축마다 "RFC/업계 가이드는 뭐라고 하는가"와 "PaintLater는 실제 코드에서 어떻게 썼는가"를 나란히 놓습니다. 특정 프레임워크가 주는 기본값 너머를 보고 싶은 주니어 백엔드 개발자를 염두에 두고 썼습니다.

## 왜 REST는 아직도 기본기인가

본론에 들어가기 전에 REST 성숙도 모델부터 간단히 짚고 넘어가겠습니다.

Leonard Richardson이 2008년에 제안한 **Richardson Maturity Model**은 REST를 네 단계로 나눕니다. Martin Fowler의 2010년 정리([Richardson Maturity Model, martinfowler.com](https://martinfowler.com/articles/richardsonMaturityModel.html))가 유명합니다.

- **Level 0** — 단일 URI에 POST 하나만 쓰는 구조 (SOAP, XML-RPC)
- **Level 1** — 리소스별로 URI는 분리하지만 HTTP 메서드는 여전히 무분별
- **Level 2** — HTTP 메서드의 의미를 준수 (GET/POST/PUT/DELETE)하고 상태 코드도 의미대로 반환
- **Level 3** — 응답 본문에 다음 액션으로 가는 링크(HATEOAS)를 포함

실무에서 "REST API"라고 부르는 대부분은 **Level 2**입니다. HTTP 인프라(캐시, 프록시, 로드밸런서)가 메서드 의미를 기본값으로 가정하기 때문에, Level 2만 제대로 맞춰도 얻는 이득이 큽니다. PaintLater 백엔드도 여기서 움직입니다.

이제 첫 번째 축부터.

## 1. 리소스와 URL 설계

### 1.1 동사는 HTTP 메서드에, 명사는 URI에

REST의 가장 기본적인 규칙은 "URI는 리소스(명사)를 식별하고, 동사는 HTTP 메서드가 담당한다"입니다. Google Cloud API Design Guide의 [Resource Names](https://cloud.google.com/apis/design/resource_names) 문서가 이를 공식적으로 권장하는 대표적인 레퍼런스입니다.

나쁜 예와 좋은 예는 이렇게 대비됩니다.

```
# 나쁨 — 동사가 URI에 박혀 있음
GET    /getMiniatures
POST   /createMiniature
DELETE /deleteMiniature/123

# 좋음 — 명사 + HTTP 메서드
GET    /miniatures
POST   /miniatures
DELETE /miniatures/123
```

PaintLater의 `MiniatureController`를 열어보면 규칙이 그대로 드러납니다.

```java file=MiniatureController.java
@RestController
@RequestMapping("/miniatures")
@RequiredArgsConstructor
public class MiniatureController {

    @GetMapping("/{id}")
    public ResponseEntity<ApiResponse<MiniatureDetailResponse>> getMiniatureDetail(...) { ... }

    @PostMapping
    public ResponseEntity<ApiResponse<MiniatureDetailResponse>> createMiniature(...) { ... }

    @PatchMapping("/{id}")
    public ResponseEntity<ApiResponse<MiniatureDetailResponse>> updateMiniature(...) { ... }

    @DeleteMapping("/{id}")
    public ResponseEntity<ApiResponse<Void>> deleteMiniature(...) { ... }
}
```

클래스 레벨 base path는 `/miniatures`(`MiniatureController.java:26`), 단건 경로는 `/{id}`입니다. 메서드명은 `getMiniatureDetail`, `createMiniature`처럼 구체적이지만, 이 동사들은 URI에 노출되지 않습니다.

### 1.2 복수형과 kebab-case

세부 컨벤션 두 가지.

**컬렉션은 복수형으로.** Google API Design Guide가 명시적으로 권하는 규칙이고, 이유는 단순합니다 — `/miniatures`가 컬렉션이고, `/miniatures/{id}`가 그 안의 한 항목이라는 관계가 언어적으로 자연스럽기 때문입니다. 단수형 `/miniature`는 컬렉션인지 항목인지 헷갈리게 만듭니다.

**두 단어 이상은 kebab-case(`-`).** URI는 `RFC 3986`에 따라 **대소문자를 구별하는 부분과 하지 않는 부분이 섞여 있는 구조**(scheme과 host는 case-insensitive, path는 case-sensitive로 간주)이기 때문에 camelCase는 환경에 따라 위험합니다. 언더스코어(`_`)는 일부 폰트에서 링크 밑줄과 구분이 어렵고, Google/Microsoft 가이드 모두 kebab-case를 권장합니다.

PaintLater의 실제 경로도 이 규칙을 따라갑니다.

```
/miniatures
/backlog-items
/miniatures/{id}/progress-logs
/miniatures/{id}/current-step
/miniatures/completed
```

`MiniatureController.java:245`의 `@PatchMapping("/{id}/current-step")`, `PublicMiniatureController.java:80`의 `@GetMapping("/{id}/progress-logs")`처럼 두 단어 이상은 전부 하이픈으로 끊어져 있습니다.

### 1.3 Path parameter vs Query parameter

RFC 3986은 URI를 **계층적 부분(path)**과 **비계층적 부분(query)**으로 나눕니다.

- **Path(§3.3)** — "이 리소스"를 특정 — `/miniatures/42`
- **Query(§3.4)** — 그 리소스 집합을 필터링/정렬/페이지네이션 — `/miniatures?page=2&size=20&status=DONE`

이 구분은 미적 취향 문제가 아니라 캐싱 동작과 직접 연결됩니다. 프록시는 query string을 캐시 키에 포함하기도, 제외하기도 하기 때문에, 식별자를 query로 빼두면 캐시 히트율을 예측하기 어려워집니다.

PaintLater는 이 경계를 의도적으로 지킵니다. `MiniatureController.java:50-80`의 `getMyMiniatures`를 보면:

```java
@GetMapping
public ResponseEntity<ApiResponse<?>> getMyMiniatures(
        @AuthenticationPrincipal CustomUserDetails userDetails,
        @RequestParam(required = false) Integer page,
        @RequestParam(required = false) Integer size,
        @RequestParam(required = false) String status,
        @RequestParam(required = false) String currentStep,
        @RequestParam(required = false, defaultValue = "createdAt") String sort,
        @RequestParam(required = false, defaultValue = "desc") String direction,
        @RequestParam(required = false) String title,
        @RequestParam(required = false) Boolean isPublic,
        @RequestParam(required = false) String category,
        @RequestParam(required = false) String subcategory
) { ... }
```

페이징, 정렬, 필터는 **전부 query parameter**입니다. `status=DONE`, `isPublic=true`, `category=historical` 같은 값이 경로 세그먼트로 새 들어가는 일은 없습니다.

### 1.4 HTTP 메서드의 Safe/Idempotent 의미

RFC 9110(HTTP Semantics, 2022)은 메서드를 두 축으로 분류합니다.

- **Safe(§9.2.1)** — 서버 상태를 변경하지 않는 읽기 전용. GET, HEAD, OPTIONS
- **Idempotent(§9.2.2)** — 같은 요청을 여러 번 보내도 한 번 보낸 것과 결과가 같음. GET, HEAD, OPTIONS, PUT, DELETE

| 메서드 | Safe | Idempotent |
|---|---|---|
| GET / HEAD / OPTIONS | ✓ | ✓ |
| PUT / DELETE | ✗ | ✓ |
| POST / PATCH | ✗ | ✗ |

이 표는 곧 **네트워크 오류가 났을 때 자동 재시도를 해도 되는가** 하는 문제로 이어집니다. POST/PATCH는 idempotent가 아니기 때문에, 타임아웃이 떴다고 무작정 재시도하면 똑같은 리소스를 두 번 만들거나 이상한 상태 전이가 생길 수 있습니다. Stripe가 [`Idempotency-Key` 헤더](https://stripe.com/docs/api/idempotent_requests)를 도입한 것이 이 문제를 풀기 위한 업계 표준 패턴입니다.

### 1.5 PUT이 아니라 PATCH, 그리고 부분 업데이트

수정 요청에는 PUT과 PATCH 두 옵션이 있습니다.

- **PUT(RFC 9110 §9.3.4)** — 리소스 전체 교체. 보내지 않은 필드는 삭제된 것으로 간주
- **PATCH(RFC 5789)** — 부분 수정. 보낸 필드만 변경, 나머지는 유지

PaintLater는 수정에는 전부 **PATCH**를 씁니다. 클라이언트(웹/모바일)가 수정 시점에 리소스 전체를 들고 있다고 장담할 수 없기 때문입니다. 썸네일만 바꾸고 싶은데 `description`이나 `category`까지 같이 보낼 책임을 클라이언트한테 떠넘기고 싶지 않다는 뜻이죠.

그런데 여기서 재미있는 설계 결정이 나옵니다. "description을 비우고 싶다(null로 설정)"와 "description을 이번 요청에서 건드리지 않는다"를 JSON 하나로 어떻게 구분할까요?

`MiniatureUpdateRequest.java:29-92`를 보면 해답이 나옵니다.

```java
public class MiniatureUpdateRequest {
    private String title;
    private String description;
    private Boolean isPublic;
    private Long projectId;
    private Integer points;
    private String category;
    private String subcategory;
    private Long thumbnailImageId;

    // "필드가 요청에 포함되었는가" 플래그
    private boolean descriptionPresent;
    private boolean projectIdPresent;
    private boolean categoryPresent;
    private boolean subcategoryPresent;

    @JsonSetter("description")
    public void setDescription(String description) {
        this.description = description;
        this.descriptionPresent = true;
    }
    // ... projectId, category, subcategory도 동일
}
```

Jackson의 `@JsonSetter`로 **"필드가 JSON에 등장했는가"** 여부를 별도 boolean에 담아둡니다. 서비스는 나중에 `request.isDescriptionPresent()`를 보고 분기해서, 플래그가 true일 때만 실제로 필드를 덮어씁니다.

RFC 7396 JSON Merge Patch가 "null은 삭제, 필드 부재는 유지"로 정리한 시맨틱을 Jackson 기본 역직렬화 안에서 구현한 셈입니다. 같은 모양의 JSON이라도 `{"description": null}`과 `{}`가 서버 입장에서는 서로 다른 의미로 처리됩니다.

### 1.6 URL prefix를 인증 경계로 쓴다

리소스 설계에서 PaintLater가 택한 구조적 결정은 URL prefix 자체를 인증 정책으로 삼는 것입니다.

`SecurityConfig.java:75-86`에 정책이 한 곳에 모여 있습니다.

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/auth/nickname").authenticated()
    .requestMatchers("/health").permitAll()
    .requestMatchers("/auth/**").permitAll()
    .requestMatchers("/public/**").permitAll()
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .anyRequest().authenticated()
)
```

읽는 법은 단순합니다. **`/public/**`으로 시작하는 경로는 permitAll, 나머지는 authenticated.** `/admin/**`은 ADMIN 롤 필수.

이 한 줄이 컨트롤러 구조에 그대로 찍힙니다. 같은 "미니어처" 리소스라도:

- 내 백로그를 CRUD하는 인증 API는 `MiniatureController`, base `/miniatures` (`MiniatureController.java:26`)
- 공개된 미니어처를 누구나 볼 수 있는 API는 `PublicMiniatureController`, base `/public/miniatures` (`PublicMiniatureController.java:24`)

두 컨트롤러가 아예 분리되어 있습니다. `MiniatureDetailResponse`와 `PublicMiniatureDetailResponse`도 별도입니다 — 공개 응답에는 `userNickname`, `likeCount`, `isOwner` 같은 "누구나 볼 수 있는 부분"만 담기고, 비공개 응답은 소유자 관점의 전체 정보를 담습니다.

왜 이렇게까지 나눴을까요? 한 컨트롤러 안에서 `if (authenticated) { ... } else { ... }`로 분기하면 권한 검사와 응답 스키마가 한 덩어리로 엉키기 때문입니다. 테스트도 까다로워지고, 공개 응답에 비공개 필드가 섞이는 사고가 나기도 쉽습니다. URL prefix를 경계로 삼아 두면 이 경계가 `SecurityConfig`의 매처와 컨트롤러 분리를 통해 자연스럽게 강제되는 효과가 있습니다.

덧붙여, `PublicMiniatureController`는 permitAll이지만 JWT 필터는 여전히 통과합니다. 토큰이 붙어 있으면 `@AuthenticationPrincipal CustomUserDetails userDetails`가 주입되므로, 같은 공개 엔드포인트가 "비로그인 방문자에게는 likeCount만", "로그인 사용자에게는 liked(내가 눌렀는지)까지" 같은 옵셔널 분기를 자연스럽게 지원합니다 (`PublicMiniatureController.java:51-53`).

### 1.7 액션 엔드포인트는 현실적인 탈출구

CRUD만으로 표현하기 어려운 비즈니스 동작들이 있습니다. "미니어처 복제", "벌크 공개 전환" 같은 것들이요. 엄격한 REST 순수주의자라면 "복제된 리소스를 POST로 만들면 되지 않나"라고 할 수 있지만, 현실적으로 다음이 더 명확합니다.

```
POST /miniatures/{id}/duplicate   (MiniatureController.java:159)
```

Google Cloud API Design Guide는 이런 패턴을 [Custom Methods](https://cloud.google.com/apis/design/custom_methods)로 공식 인정합니다. Stripe API의 [`POST /v1/payment_intents/{id}/cancel`](https://stripe.com/docs/api/payment_intents/cancel)도 같은 패턴의 실제 예입니다.

PaintLater는 이 탈출구를 두 가지 용도로 씁니다.

**하위 속성 단건 업데이트**. `PATCH /miniatures/{id}/current-step` — `currentStep` 하나만 바꾸는 전용 엔드포인트. 바디는 `MiniatureStepUpdateRequest`의 필드 하나뿐입니다 (`MiniatureStepUpdateRequest.java:14`). 전체 PATCH로 할 수도 있지만 **"이 필드만 바꾸는 의도"를 URL 수준에서 드러내는 쪽**을 택했습니다.

**벌크 작업**. `DELETE /miniatures/bulk-delete`, `PATCH /miniatures/bulk-visibility`처럼 컬렉션 단위 조작을 위한 전용 경로. 클라이언트가 "여러 개 한 번에 처리"의 의도를 실수로 숨기지 못하게 강제하는 효과가 있습니다.

교과서에는 잘 안 나오지만 실무에선 광범위하게 쓰이는 패턴이고, 코드에 의도를 드러내는 데 크게 기여합니다.

## 2. 상태 코드와 에러 응답

두 번째 축으로 넘어갑니다. REST API의 상태 코드는 겉보기에 단순해 보이지만, 실제로는 가장 많은 사람이 가장 많이 틀리는 영역이기도 합니다.

### 2.1 RFC 9110의 5개 클래스

상태 코드의 의미론은 [RFC 9110 §15](https://www.rfc-editor.org/rfc/rfc9110#section-15)에 정리되어 있습니다. 다섯 개의 클래스가 있고, 각각 의미가 다릅니다.

- **1xx Informational** — 요청 수신, 처리 중
- **2xx Successful** — 성공
- **3xx Redirection** — 추가 조치 필요
- **4xx Client Error** — 클라이언트 책임
- **5xx Server Error** — 서버 책임

실무에서 가장 자주 혼용되는 몇 가지만 정리하면:

| 코드 | RFC 9110 | 의미 |
|---|---|---|
| 200 OK | §15.3.1 | 성공, 본문 있음 |
| 201 Created | §15.3.2 | 새 리소스 생성됨. `Location` 헤더 권장 (§10.2.2) |
| 204 No Content | §15.3.5 | 성공, 본문 없음 |
| 400 Bad Request | §15.5.1 | 문법/형식 오류 |
| 401 Unauthorized | §15.5.2 | **미인증** (이름이 혼란스럽지만 인증이 필요하다는 뜻) |
| 403 Forbidden | §15.5.4 | 인증은 됐으나 **권한 없음** |
| 404 Not Found | §15.5.5 | 리소스 없음 |
| 409 Conflict | §15.5.10 | 현재 상태와 요청의 충돌 |
| 422 Unprocessable Content | §15.5.21 | 문법은 OK지만 의미론적으로 처리 불가 |
| 429 Too Many Requests | RFC 6585 §4 | Rate limit 초과 |

401과 403은 구분해서 사용해야 합니다. 401은 "인증이 필요하다"는 의미이고, 403은 "인증은 끝났지만 이 리소스에 대한 권한이 없다"는 의미입니다. 프론트 입장에서는 두 경우에 필요한 동작이 다릅니다. 401이면 로그인 페이지로 보내고, 403이면 에러 토스트를 띄우는 식입니다. 두 코드를 섞어서 내려주면 UX 분기가 깨집니다.

### 2.2 ErrorCode enum을 단일 소스로

이 문제를 PaintLater가 풀어가는 방식은 다음과 같습니다.

에러 응답을 설계할 때 흔히 빠지는 함정 하나는, 컨트롤러마다 상태 코드를 직접 박아 두는 스타일입니다. `return ResponseEntity.status(404).body(...)`가 여기저기 흩어지면, 같은 "리소스 없음"이 어느 엔드포인트에서는 404, 어느 엔드포인트에서는 400으로 돌아오는 일이 금방 벌어집니다.

PaintLater는 이 문제를 `ErrorCode` enum 하나에 전부 몰아넣어서 풉니다 (`common/exception/ErrorCode.java`).

```java file=ErrorCode.java
public enum ErrorCode {
    // 공통 (E1xxx)
    INTERNAL_SERVER_ERROR("E1000", "서버 내부 오류가 발생했습니다.", HttpStatus.INTERNAL_SERVER_ERROR),
    INVALID_INPUT_VALUE("E1001", "입력값이 올바르지 않습니다.", HttpStatus.BAD_REQUEST),
    METHOD_NOT_ALLOWED("E1002", "허용되지 않은 메서드입니다.", HttpStatus.METHOD_NOT_ALLOWED),
    RESOURCE_NOT_FOUND("E1003", "요청하신 리소스를 찾을 수 없습니다.", HttpStatus.NOT_FOUND),

    // 인증/인가 (E2xxx)
    UNAUTHORIZED("E2000", "인증이 필요합니다.", HttpStatus.UNAUTHORIZED),
    ACCESS_DENIED("E2001", "접근 권한이 없습니다.", HttpStatus.FORBIDDEN),
    INVALID_TOKEN("E2002", "유효하지 않은 토큰입니다.", HttpStatus.UNAUTHORIZED),
    EXPIRED_TOKEN("E2003", "만료된 토큰입니다.", HttpStatus.UNAUTHORIZED),
    RATE_LIMIT_EXCEEDED("E2005", "요청이 너무 많습니다.", HttpStatus.TOO_MANY_REQUESTS),

    // 미니어처 (E4xxx)
    MINIATURE_NOT_FOUND("E4000", "미니어처를 찾을 수 없습니다.", HttpStatus.NOT_FOUND),
    MINIATURE_ACCESS_DENIED("E4001", "미니어처 접근 권한이 없습니다.", HttpStatus.FORBIDDEN),
    // ...
    ;

    private final String code;
    private final String message;
    private final HttpStatus httpStatus;
}
```

enum 한 항목이 `(code 문자열, 메시지, HttpStatus)` 세 가지를 한 덩어리로 들고 있는 구조입니다. 이 구조가 왜 효과적인지는 다음 두 파일에서 확인할 수 있습니다.

**`BusinessException.java`**
```java
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }
}
```

**`GlobalExceptionHandler.java:28-40`**
```java
@ExceptionHandler(BusinessException.class)
public ResponseEntity<ApiResponse<Void>> handleBusinessException(BusinessException e) {
    ErrorCode errorCode = e.getErrorCode();
    log.warn("BusinessException: {} - {}", errorCode.getCode(), errorCode.getMessage());
    return ResponseEntity
            .status(errorCode.getHttpStatus())   // ← enum이 상태 코드를 결정
            .body(ApiResponse.error(errorCode));
}
```

서비스 레이어에는 HTTP 상태 코드가 등장하지 않습니다. `throw new BusinessException(ErrorCode.MINIATURE_NOT_FOUND)`만 던지면, 전역 핸들러가 enum에서 `HttpStatus.NOT_FOUND`를 꺼내 404 응답으로 변환해 줍니다.

실제 사용 예는 서비스 곳곳에 반복됩니다.

```java file=MiniatureService.java
Miniature miniature = miniatureRepository.findById(miniatureId)
        .orElseThrow(() -> new BusinessException(ErrorCode.MINIATURE_NOT_FOUND));
validateOwnership(miniature, userId);
```

```java file=MiniatureService.java
private void validateOwnership(Miniature miniature, Long userId) {
    if (!miniature.getUserId().equals(userId)) {
        log.warn("미니어처 접근 권한 없음: miniatureId={}, ownerId={}, requesterId={}",
                miniature.getId(), miniature.getUserId(), userId);
        throw new BusinessException(ErrorCode.MINIATURE_ACCESS_DENIED);
    }
}
```

`MiniatureService.java:425`는 `findById().orElseThrow()` 관용구, `MiniatureService.java:577-583`은 소유권 검증입니다. 상태 코드가 enum 바깥에 등장하는 경우는 없습니다.

이 구조가 주는 이점은 다음과 같습니다.

1. **상태 코드 정책이 한 파일에 모인다.** 새 403이 필요하면 `ErrorCode`에 항목만 하나 추가하면 끝. 핸들러는 건드릴 일이 없습니다.
2. **프론트가 `code`로 분기할 수 있다.** `E2003`(EXPIRED_TOKEN)을 받으면 토큰 갱신을 시도하고, `E2002`(INVALID_TOKEN)면 즉시 로그아웃. 상태 코드는 둘 다 401이라 HTTP 레벨로는 구분이 안 되지만, 애플리케이션 에러 코드가 있으면 나눌 수 있습니다. ([인증 포스트에서 이 분기가 왜 중요한지 다뤘습니다](/posts/jwt-token-queue-pattern))
3. **`_NOT_FOUND`와 `_ACCESS_DENIED`를 짝으로 관리한다.** `MINIATURE_NOT_FOUND`(E4000, 404)와 `MINIATURE_ACCESS_DENIED`(E4001, 403)가 같은 도메인 접두사 아래 나란히 선언돼 있어서, 404와 403을 헷갈릴 여지가 enum 네이밍 단계에서 없어집니다.

### 2.3 응답 봉투는 `ApiResponse<T>` 하나

에러뿐 아니라 **성공 응답도 한 가지 봉투**로 통일됩니다 (`common/dto/ApiResponse.java`).

```java file=ApiResponse.java
@JsonInclude(JsonInclude.Include.NON_NULL)
public class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;
    private ErrorDetail error;
    private Instant timestamp;

    @JsonInclude(JsonInclude.Include.NON_NULL)
    public static class ErrorDetail {
        private String code;
        private String message;
        private String detail;
    }
}
```

`@JsonInclude(NON_NULL)` 덕에 **성공 응답에는 `error`가, 실패 응답에는 `data`가 직렬화되지 않습니다.** 같은 클래스 하나가 두 모양을 담는 셈입니다.

실제로 내려가는 JSON 예시:

```json
// 성공 (GET /miniatures/42)
{
  "success": true,
  "data": {
    "id": 42,
    "title": "Necron Overlord",
    "progress": 60,
    "backlogItems": [ ... ]
  },
  "timestamp": "2026-04-13T12:34:56.789Z"
}

// 실패 (GET /miniatures/99999 — 없는 ID)
{
  "success": false,
  "error": {
    "code": "E4000",
    "message": "미니어처를 찾을 수 없습니다."
  },
  "timestamp": "2026-04-13T12:34:56.789Z"
}

// 검증 실패 (POST /miniatures 바디 누락)
{
  "success": false,
  "error": {
    "code": "E1001",
    "message": "입력값이 올바르지 않습니다.",
    "detail": "title: 제목은 필수입니다., points: 포인트는 0 이상이어야 합니다."
  },
  "timestamp": "2026-04-13T12:34:56.789Z"
}
```

> **RFC 9457 Problem Details와의 관계**
>
> RFC 9457(2023, RFC 7807 대체)은 HTTP API 에러 응답을 위한 표준 포맷을 정의합니다. Content-Type은 `application/problem+json`, 표준 필드는 `type`(문제 유형 URI), `title`, `status`, `detail`, `instance`입니다.
>
> PaintLater의 `ApiResponse`는 Problem Details를 **따르지 않습니다.** 대신 `success` 플래그 + 도메인 고유 `code` 문자열을 쓰는 자체 포맷입니다. 장점은 성공과 실패를 같은 봉투로 통일할 수 있다는 점, 단점은 업계 표준 클라이언트 라이브러리와의 호환성이 떨어진다는 점. 내부 서비스에서는 자체 포맷이 간단하고, 외부에 공개하는 API에는 Problem Details가 더 적합하다는 판단이 실무적 트레이드오프입니다.

### 2.4 검증 실패도 같은 흐름을 탄다

`@Valid`를 붙인 DTO의 검증 실패도 같은 봉투로 내려갑니다. `GlobalExceptionHandler.java:45-60`:

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ApiResponse<Void>> handleValidationException(MethodArgumentNotValidException e) {
    String detail = e.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .collect(Collectors.joining(", "));

    ErrorCode errorCode = ErrorCode.INVALID_INPUT_VALUE;
    return ResponseEntity
            .status(errorCode.getHttpStatus())
            .body(ApiResponse.error(errorCode, detail));
}
```

`@Valid @RequestBody MiniatureCreateRequest` (`MiniatureController.java:104`)에서 검증 실패가 나면 Spring이 `MethodArgumentNotValidException`을 던지고, 위 핸들러가 필드별 메시지를 이어붙여 `detail`에 담아 400 + `E1001`로 내려줍니다. 클라이언트는 정확히 어떤 필드가 왜 실패했는지 파싱할 수 있습니다.

쿼리 파라미터 검증(`@Min`, `@Max`) 실패 경로인 `ConstraintViolationException`도 같은 방식으로 매핑됩니다 (`GlobalExceptionHandler.java:65-80`). 결과적으로 **모든 4xx는 같은 모양**으로 내려갑니다.

## 3. DTO 경계 설계

마지막 축. 앞의 두 축이 "인터페이스의 바깥"을 다룬다면, 이 축은 "인터페이스와 내부 도메인 모델 사이의 벽"을 다룹니다.

### 3.1 JPA 엔티티를 절대 노출하지 않는 이유

백엔드에서 가장 흔한 초기 실수 중 하나는 **JPA `@Entity`를 컨트롤러에서 그대로 반환하는 것**입니다. 동작은 하지만, 네 가지 문제가 곧 따라옵니다.

1. **Lazy loading 예외.** Jackson이 직렬화하면서 지연 로딩된 연관을 건드리면 `LazyInitializationException`이 터지거나, 의도치 않은 N+1 쿼리가 뱃치로 나갑니다. Vlad Mihalcea와 Thorben Janssen이 블로그에서 반복해서 경고하는 문제입니다.
2. **순환 참조.** 양방향 연관(`@OneToMany` + `@ManyToOne`)이 있으면 직렬화가 무한 루프에 빠져 `StackOverflowError`로 끝납니다.
3. **내부 필드 노출.** `version`(낙관적 잠금), 감사용 `createdBy`, `deleted` 플래그 같은 것들이 클라이언트에 그대로 흘러나갑니다.
4. **API-스키마 결합.** DB 컬럼명을 바꾸면 API 계약이 깨집니다. 두 관심사가 엉켜 있으면 스키마 마이그레이션이 "프론트 릴리스와 동시에 해야 하는 일"이 돼버립니다.

해결책은 단순합니다. **컨트롤러는 엔티티를 절대 반환하지 않는다, DTO만 반환한다.**

PaintLater는 이 원칙을 예외 없이 지킵니다. 네 개 컨트롤러(`MiniatureController`, `PublicMiniatureController`, `GalleryController`, `BacklogItemController`)의 모든 핸들러 반환 타입을 확인해 보면, JPA 엔티티인 `Miniature`나 `BacklogItem`이 반환 경로에 노출되는 지점은 찾을 수 없습니다. 모든 응답은 `*Response` DTO를 거쳐 `ApiResponse<T>` 봉투로 감싸져 나갑니다.

### 3.2 Request와 Response DTO를 분리한다

DTO를 쓰더라도 요청과 응답이 하나의 DTO를 공유하는 선택지가 있습니다. 이것도 피해야 할 안티패턴입니다.

- 생성 요청에는 `id`가 없지만 응답에는 있어야 한다 → `@JsonIgnore`로 한쪽만 숨기면서 오염
- 요청용 검증 어노테이션(`@NotBlank`, `@Size`)이 응답 DTO에까지 따라붙는다
- 응답에만 필요한 계산 필드(`progress`, `thumbnailUrl`)가 요청 쪽에도 섞인다

PaintLater의 DTO 네이밍 규칙은 이 문제를 아예 발생하지 않게 만듭니다.

- `*Request` — 요청 전용 (`MiniatureCreateRequest`, `MiniatureUpdateRequest`, `BacklogItemUpdateRequest`)
- `*Response` — 응답 전용 (`MiniatureResponse`, `MiniatureDetailResponse`, `GalleryRankingResponse`)
- 역할 수식은 prefix/suffix로 — `Detail`, `Summary`, `Page`, `Public`, `Bulk*`

`*Dto` 같은 모호한 접미사는 사용되지 않습니다.

### 3.3 목록용과 상세용을 나눈다

응답 DTO를 한 번 더 쪼갠 것도 중요한 결정입니다. 같은 미니어처라도 목록 뷰와 상세 뷰가 원하는 데이터는 서로 다릅니다.

```
MiniatureResponse                (목록용)
├── id, title, isPublic
├── progress, currentStep
├── thumbnailUrl 세 종류
├── projectId / projectName
├── points, category, subcategory
└── createdAt, updatedAt

MiniatureDetailResponse          (상세용)
├── (위 필드 전부)
├── description               ← 상세에만
├── backlogItems: List<BacklogItemResponse>   ← 상세에만
└── linkedGalleryPosts        ← 상세에만
```

목록 API가 100개의 미니어처를 돌려줄 때 각 항목의 `backlogItems`와 `description`까지 다 담으면 페이로드가 폭발합니다. 클라이언트가 "카드 리스트"만 그리는데 서버가 "상세 페이지 전체"를 보내는 셈이니까요.

흔히 over-fetching이라 부르는 문제고, GraphQL이 쿼리 언어 차원에서 푼 바로 그 문제이기도 합니다. REST에서는 **뷰별로 DTO를 나누는 것**이 가장 현실적인 답입니다. JSON:API 스펙은 [Sparse Fieldsets](https://jsonapi.org/format/#fetching-sparse-fieldsets)로 `?fields=id,title` 문법을 표준화해뒀지만, 대부분의 Spring 프로젝트는 PaintLater처럼 DTO 분리로 해결합니다.

### 3.4 `of(...)` 팩토리 패턴과 계산값 주입

엔티티에서 DTO로 변환하는 코드는 어디서 살아야 할까요? 세 가지 선택지가 있습니다.

- **수동 정적 팩토리** — `MiniatureResponse.of(entity, ...)` 같은 메서드를 DTO 자체에 정의
- **[MapStruct](https://mapstruct.org/)** — 어노테이션 프로세서로 컴파일 타임에 매핑 코드 생성. 리플렉션 없이 순수 Java 코드로 생성되므로 수동 작성과 동등한 성능
- **ModelMapper** — 런타임 리플렉션, 설정 간단하지만 성능과 타입 안전성이 떨어짐

PaintLater가 선택한 방식은 수동 정적 팩토리입니다. MapStruct의 장점(보일러플레이트 감소, 컴파일 타임 안전)이 분명하지만, 수동 팩토리에만 있는 이점 때문에 이쪽을 택했습니다. 이유는 아래 코드에서 드러납니다.

`MiniatureDetailResponse.java:47-73`:

```java
public static MiniatureDetailResponse of(
        Miniature miniature,
        int progress,
        List<BacklogItemResponse> backlogItems,
        String projectName,
        String thumbnailUrl,
        List<LinkedGalleryPost> linkedGalleryPosts
) {
    return MiniatureDetailResponse.builder()
            .id(miniature.getId())
            .title(miniature.getTitle())
            .description(miniature.getDescription())
            .isPublic(miniature.getIsPublic())
            .progress(progress)                     // ← 엔티티에 없는 계산값
            .backlogItems(backlogItems)             // ← 서비스가 미리 로드
            .projectId(miniature.getProjectId())
            .projectName(projectName)               // ← 별도 쿼리 결과
            .points(miniature.getPoints())
            .category(miniature.getCategory())
            .subcategory(miniature.getSubcategory())
            .thumbnailImageId(miniature.getThumbnailImageId())
            .thumbnailUrl(thumbnailUrl)             // ← 사전 계산
            .createdAt(miniature.getCreatedAt())
            .updatedAt(miniature.getUpdatedAt())
            .linkedGalleryPosts(linkedGalleryPosts) // ← 별도 쿼리 결과
            .build();
}
```

여기서 주목할 부분이 있습니다.

`miniature.getX()`로 꺼내는 값은 엔티티가 자체적으로 보유한 필드뿐입니다. `progress`(진행률 계산), `backlogItems`(자식 컬렉션), `projectName`(다른 테이블 조회), `thumbnailUrl`(R2 경로 변환), `linkedGalleryPosts`(갤러리 역참조)처럼 연관을 따라가야 나오는 값은 모두 서비스 레이어에서 계산해 인자로 넘겨주는 구조입니다.

이 규약 덕분에 DTO 변환 시점에 JPA 연관 탐색이 일어나지 않습니다. 지연 로딩된 `miniature.getBacklogItems()`를 변환 코드에서 건드릴 일이 구조적으로 없어지는 셈입니다.

서비스 레이어는 명시적으로 필요한 것만 로드하고, 변환 함수에 전부 넘깁니다. `MiniatureService.java:410-416`:

```java
String projectName = getProjectName(miniature.getProjectId());
Map<Long, String> thumbnailMap = thumbnailResolver.buildThumbnailMap(List.of(miniature), false);
String thumbnailUrl = thumbnailMap.get(miniature.getId());

return MiniatureDetailResponse.of(
    miniature, progress, backlogItemResponses, projectName, thumbnailUrl, getLinkedGalleryPosts(miniatureId)
);
```

`of()`가 인자를 많이 받도록 강제한 구조가 사실상 코드 리뷰 체크리스트 역할을 합니다. 새 필드를 추가하는 개발자가 "이 값은 어디서 가져와야 하지?"를 한 번 짚고 넘어가게 됩니다. MapStruct의 자동화 편의를 포기한 대신 얻은 안전장치라고 볼 수 있습니다.

같은 패턴이 `MiniatureResponse.of(...)`, `PublicMiniatureResponse.of(...)`, `PublicMiniatureDetailResponse.of(...)`, `BacklogItemResponse.from(...)` 전부에 반복됩니다. `of()`는 복수 인자, `from()`은 엔티티 하나만 받을 때 — 암묵적 네이밍 규칙까지 지켜집니다.

### 3.5 Bean Validation으로 경계에서 검증한다

입력 DTO는 받는 즉시 검증합니다. Jakarta Validation 3.0(구 JSR 380)의 표준 어노테이션이 이 역할을 합니다.

`MiniatureCreateRequest.java`:
```java
public class MiniatureCreateRequest {
    @NotBlank(message = "제목은 필수입니다.")
    @Size(max = 200, message = "제목은 200자 이하여야 합니다.")
    private String title;

    @Size(max = 1000, message = "설명은 1000자 이하여야 합니다.")
    private String description;

    @Min(value = 0, message = "포인트는 0 이상이어야 합니다.")
    private Integer points;

    @Size(max = 30)
    private String category;

    @Size(max = 100)
    private String subcategory;

    // ...
}
```

컨트롤러에서는 `@Valid`를 붙여 검증을 걸어줍니다.

```java
@PostMapping
public ResponseEntity<ApiResponse<MiniatureDetailResponse>> createMiniature(
        @AuthenticationPrincipal CustomUserDetails userDetails,
        @Valid @RequestBody MiniatureCreateRequest request     // ← @Valid
) { ... }
```

Spring Boot의 `spring-boot-starter-validation`이 Hibernate Validator(레퍼런스 구현체)를 포함하고, `@Valid`가 실패하면 위에서 본 `GlobalExceptionHandler`의 `MethodArgumentNotValidException` 핸들러로 빠져 400 + `E1001`로 내려갑니다.

자주 쓰는 어노테이션 정리:

| 어노테이션 | 적용 | 의미 |
|---|---|---|
| `@NotNull` | 모든 타입 | null 불허 (빈 문자열은 통과) |
| `@NotBlank` | String | null, 빈 문자열, 공백 문자열 전부 불허 |
| `@NotEmpty` | String, Collection | null/빈 불허 |
| `@Size(min, max)` | String, Collection | 길이 범위 |
| `@Min` / `@Max` | 숫자 | 값 범위 |
| `@Email` | String | 이메일 형식 |
| `@Pattern(regexp)` | String | 정규식 |
| `@Valid` | 중첩 객체 | 검증 cascade |

실무에서 가장 헷갈리는 건 `@NotNull` vs `@NotBlank` vs `@NotEmpty`입니다. 문자열을 받을 때 `@NotNull`만 쓰면 `""`(빈 문자열)이 통과해서 DB에 빈 레코드가 쌓입니다. 문자열에는 기본적으로 `@NotBlank`를 쓰는 편이 안전합니다.

### 3.6 페이지네이션 응답을 평탄화한다

Spring Data의 `Page<T>`는 편리하지만, 컨트롤러에서 그대로 반환하면 내부 구현체인 `PageImpl`이 그대로 직렬화됩니다.

```json
{
  "content": [ ... ],
  "pageable": {
    "pageNumber": 0,
    "pageSize": 20,
    "sort": { ... },
    "offset": 0,
    "paged": true,
    "unpaged": false
  },
  "totalElements": 150,
  "totalPages": 8,
  "last": false,
  "first": true,
  "numberOfElements": 20,
  "empty": false
}
```

Spring의 내부 구조가 그대로 API 계약으로 굳어지는 상태입니다. Spring Data 버전을 올려 `PageImpl`이 바뀌면 API 계약도 따라서 변합니다. 구현과 인터페이스가 한 덩어리로 엉킨 전형적인 안티패턴입니다.

PaintLater는 `Page<T>`를 직접 노출하지 않습니다. 대신 프로젝트 전용 `*PageResponse` DTO로 한 번 평탄화합니다.

`MiniaturePageResponse.java:25-35`:
```java
public static MiniaturePageResponse from(Page<MiniatureResponse> page) {
    return MiniaturePageResponse.builder()
            .content(page.getContent())
            .page(page.getNumber())
            .size(page.getSize())
            .totalElements(page.getTotalElements())
            .totalPages(page.getTotalPages())
            .hasNext(page.hasNext())
            .hasPrevious(page.hasPrevious())
            .build();
}
```

결과 JSON:
```json
{
  "content": [ ... ],
  "page": 0,
  "size": 20,
  "totalElements": 150,
  "totalPages": 8,
  "hasNext": true,
  "hasPrevious": false
}
```

필요한 필드만 남고, Spring 내부 디테일(`pageable`, `sort`, `empty`, `numberOfElements`)은 응답에서 빠집니다. 이 구조에서는 Spring Data 버전이 올라가도 API 계약이 영향을 받지 않습니다.

## 교훈

세 축을 정리하고 나면 공통된 패턴이 보입니다.

**1. 프레임워크 기본값을 그대로 노출하지 않는다.**

JPA 엔티티를 그대로 반환하면 Hibernate의 내부 구조가 API로 새어 나가고, `Page<T>`를 그대로 반환하면 Spring Data의 내부 구조가 API로 새어 나갑니다. 두 경우 모두 동작은 정상적으로 하지만, 프레임워크와 API 계약이 강하게 결합되는 접근입니다. 얇은 변환 레이어를 한 겹 끼워 두기만 해도 이 결합을 끊을 수 있습니다.

**2. 정책을 단일 파일에 모은다.**

인증 정책은 `SecurityConfig`의 `authorizeHttpRequests` 한 블록에, 에러 정책은 `ErrorCode` enum 한 파일에, 응답 봉투 포맷은 `ApiResponse` 한 클래스에 모여 있습니다. 새로운 규칙이 필요할 때 수정할 파일이 명확해집니다. 규약이 코드 전체에 흩어져 있으면 시간이 지날수록 일관성을 유지하기 어려워집니다.

**3. 컴파일러가 강제할 수 있는 규약으로 바꾼다.**

- 엔티티 대신 DTO만 반환 → 타입 시스템이 강제
- URL prefix 기반 인증 분리 → `SecurityConfig`의 매처가 강제
- `of()` 팩토리에 계산값 인자 → 컴파일 에러로 누락 차단
- `@Valid` + Bean Validation → 어노테이션이 검증 강제

문서나 컨벤션으로만 남은 규약은 시간이 지나면 잊히기 쉽습니다. 컴파일러가 에러를 내는 구조로 바꿔 두면 사람이 매번 기억하고 있지 않아도 됩니다.

**4. 프론트와의 계약 언어를 하나 만든다.**

`ErrorCode` enum의 `E2003`, `E4001` 같은 코드는 단순한 디버깅 힌트가 아닙니다. 프론트는 이 코드를 보고 "토큰 갱신을 시도할지, 로그아웃할지, 에러 토스트를 띄울지"를 결정하거든요. HTTP 상태 코드(`401`) 하나로는 부족한 구체성을 애플리케이션 에러 코드가 메꿔주는 셈입니다.

---

REST Level 3 HATEOAS 같은 형식 준수는 실무에서 거의 쓸 일이 없습니다. 다만 이 글에서 다룬 Level 2의 세 축(리소스 설계, 상태 코드, DTO 경계)은 프레임워크가 기본으로 제공하지 않는 영역이고, API 수가 10개에서 100개로 늘어날 때 가장 먼저 일관성이 흐트러지는 지점이기도 합니다.

프레임워크가 답을 주지 않는 영역에 규약을 직접 박아 넣는 작업이, 결국 실전 REST API 설계의 상당 부분을 차지한다고 생각됩니다.
